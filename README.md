# mas-gitops-config (IBM-aligned, hub-and-spoke)

ONE branch holds EVERY cluster. The single Account Root Application on the management cluster
discovers each `mas/<cluster>/` directory via its `<account>/*` glob and deploys that cluster's
workloads to its Target Cluster via `cluster.url`.

```
base/         # templates (edit these, not the generated output)
  cluster/    # -> mas/<cluster>/*.yaml          (cluster-scope, merged by merge-key mas/<cluster>)
  instance/   # -> mas/<cluster>/<instance>/*.yaml (instance-scope)
  vault/      # complete HashiCorp Vault loader template
envs/         # one tiny <cluster>.env per cluster (the ~6 values that differ)
mas/          # GENERATED by render.py - committed so ArgoCD can read it
vault/        # GENERATED per-cluster loaders
render.py / render.sh
```

## Render everything (single branch)
```bash
./render.sh            # renders all envs/*.env into mas/
git add mas/ && git commit -m "config" && git push    # one branch, e.g. main
```

## Notes
- No branch-per-env. Every cluster is a directory on the same branch.
- Secrets live in ONE HashiCorp Vault under `secret/<account>/<cluster>/...`. AVP on the
  management cluster resolves them centrally, so Target Clusters need no Vault of their own.
- External Oracle JDBC: `jdbc_ca_pem` is active because `ibm-jdbc-config` forces `sslEnabled: true`.
- Only `system` config scope is supported by MAS GitOps except for JDBC (which allows all scopes);
  these templates use `system` for SLS/Mongo/JDBC, which is compliant.
